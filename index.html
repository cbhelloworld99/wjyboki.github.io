状态机

状态机全称: 有限状态机(Finite State Machine)

状态机是使不同状态之间相互转换以及在状态发生改变时触发相应事件的一种机制

状态机核心: 不原地等待,只记住进度

状态机的四要素

  • 当前状态
  • 触发条件: 状态发生改变的条件
  • 事件: 状态发生改变时产生的相应动作
  • 下一状态

注意: 状态是一种持续性的 事件或动作是间隔的 状态改变产生事件 事件完成后 状态依然持续

使用状态机优化按键扫描

按键扫描常用实现方法:

  • 轮询法: 定期检查按键状态
  • 中断法: 按键按下时触发中断
  • 状态机法: 使用状态机模型优化按键检测

对比使用状态机法和其他的区别

  • 资源占用:
    • 轮询法: 在循环中一直占用CPU资源
    • 中断法: 只有在触发中断时(时间产生后跳转执行回调)才占用CPU资源,但多个按键需要多个中断向量
    • 状态机法: 使用状态机法,只需使用一个定时器或者操作系统中的任务调度器去实现按键检测即可,同样占用少量CPU资源
  • 执行效率:
    • 轮询法: 效率极低且触发不准确
    • 中断法: 效率较高,但单个中断方式不支持长按
    • 状态机法: 效率较高,支持长按
  • 按键消抖:
    • 轮询法: 依赖延时函数,会造成阻塞
    • 中断法: RC硬件消抖,成本高 或结合定时器 实现稍复杂 无阻塞
    • 状态机法: 消抖简单,如果使用20ms定时器则可在状态转变过程中就实现了消抖 无阻塞

映射表的概念以及运用

映射表的概念

映射表的本质 : 将一个值(输入) 映射到另一个值(输出)的一种数据结构 通过查表代替条件判断

输入(索引) → [映射表] → 输出(值)

使用映射表的优势

  • 解耦 : 将输入和输出解耦 方便以后对输入进行扩展 增加扩展性
  • 执行效率高: O(1)时间复杂度
  • 提高代码可读性 : 通过查表代替条件判断 减少代码量

映射表的应用

应用一 使用映射表进行按键值与按键事件的解耦

前面已经介绍过用状态机的方式处理按键扫描 做到了驱动层和业务层的解耦–> 底层驱动只关注状态的变化 上层业务只关注事件的触发

但在业务层还有不同的按键类型去触发不同的按键事件
比如: 一个普通的消费电子功能时钟 有时间设置按键 闹钟设置按键 闹钟开关按键 星期闹钟按键 上调下调等这些代表不同功能的按键类型 每一个按键类型都可以对应一个或多个按键事件 如 时间设置进行单击短按是切换时间格式,而长按是进入时间设置界面 根据之前的逻辑我们需要key_proc函数中在不同事件中对不同按键类型进行判断并执行相对应的处理函数,代码冗长每个按键都需要使用if判断或使用case分支,执行效率底下需要逐个比较并且十分不利于扩展

现在我们可以使用映射表来解耦按键类型和按键事件

首先,按键ID是唯一的,每一个ID对应着一种按键类型,每一种按键类型对应多种按键事件(多个按键值)
如: KEY_UP_ID —> KEY_UP —> KEY_UP / KEY_UP_LONG / KEY_UP_REPEAT

优劣对比

传统方式

优势:

  • 可读性高 –> 将大量嵌套进行解耦 将数据与逻辑分开
  • 可维护性好 –> 若要修改按键功能 只需修改映射表中数据 无需修改主代码逻辑
  • 扩展性佳 –> 若要添加新的按键功能 只需在映射表中添加新的数据 无需修改主代码的逻辑

劣势: 代码量相对较大，需要额外的映射表结构 但在巨大优势面前，劣势不值一提
